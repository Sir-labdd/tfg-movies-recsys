## 7.3.2 Diseño de la base de datos y carga de datos

> *La subsección anterior dejó cuatro archivos CSV limpios en `data/processed/`, listos para ser cargados en una base de datos relacional. Esta subsección documenta el diseño del esquema PostgreSQL que los alberga, el mecanismo de migraciones que lo versiona, y los scripts de carga que transfieren los datos procesados a las tablas definitivas.*

### 7.3.2.1 Planificación

El diseño de la base de datos y la carga de datos constituyen los bloques B3 y B4 de la planificación temporal expuesta en el apartado 7.1. La descomposición sigue el orden natural de dependencias: primero las extensiones y el esquema (sin datos), después la infraestructura de migraciones que lo versiona, y finalmente los scripts de carga que pueblan las tablas.

**Sub-paso B3.1 — Infraestructura de contenedores.** Configurar un fichero `docker-compose.yml` para levantar un contenedor de PostgreSQL 16 con la extensión pgvector preinstalada, exponiendo el puerto 5432 en desarrollo local. El volumen de datos se monta de forma persistente para sobrevivir a reinicios del contenedor.

**Sub-paso B3.2 — Esquema relacional.** Diseñar e implementar como ficheros SQL las tablas del esquema, organizadas en cinco migraciones temáticas: extensiones, películas y colecciones, personas y créditos, categorías (géneros, compañías productoras, países, idiomas, palabras clave) con sus tablas de unión, y identificadores externos. Cada tabla sigue las convenciones de nombrado PostgreSQL (snake_case, nombres en plural para entidades, singular para tablas de unión con prefijo de las dos entidades).

**Sub-paso B3.3 — Sistema de migraciones.** Implementar un script Python (`07_migrate.py`) que aplica migraciones SQL de forma secuencial e idempotente, registrando cada migración aplicada en una tabla `schema_migrations` con su nombre, marca temporal y duración. Las migraciones posteriores (índices, vistas, evolución del esquema de embeddings) se añaden siguiendo el mismo mecanismo.

**Sub-paso B4.1 — Carga de películas y colecciones.** Implementar un script que lee `movies.csv`, extrae las colecciones únicas del campo JSON `belongs_to_collection`, las carga en la tabla `collections`, y después carga las películas en la tabla `movies` con la referencia a su colección.

**Sub-paso B4.2 — Carga de categorías.** Implementar un script que, para cada una de las cinco familias de categorías (géneros, compañías productoras, países de producción, idiomas hablados y palabras clave), parsea el campo JSON correspondiente, acumula las entidades únicas, las inserta en la tabla maestra con `ON CONFLICT DO NOTHING`, y carga las filas de la tabla de unión en bloque mediante `COPY`.

**Sub-paso B4.3 — Carga de créditos.** Implementar un script que parsea los campos JSON `cast` y `crew` de `credits.csv`, inserta las personas únicas en la tabla `people`, y carga las relaciones en las tablas `movie_cast` y `movie_crew`.

**Sub-paso B4.4 — Carga de identificadores externos.** Implementar un script que carga las equivalencias entre identificadores TMDB, IMDb y MovieLens desde `links.csv` en la tabla `external_ids`.

**Sub-paso B4.5 — Orquestación de la carga.** Implementar un script orquestador (`12_run_load_pipeline.py`) que ejecuta los cuatro scripts de carga en secuencia, siguiendo el mismo patrón del orquestador de limpieza.

### 7.3.2.2 Investigación

**PostgreSQL con pgvector frente a alternativas.** La elección de PostgreSQL como gestor de base de datos se realizó en la fase de planificación (apartado 7.1) por dos motivos. Primero, PostgreSQL es la base de datos relacional de código abierto más madura y mejor documentada, con soporte nativo para tipos de datos avanzados (arrays, JSON, fechas con rango) que se adaptan naturalmente a la estructura del dataset. Segundo, la extensión **pgvector** permite almacenar vectores de alta dimensionalidad y realizar búsquedas de vecinos más cercanos directamente en SQL, sin necesidad de un servicio vectorial externo como Pinecone, Weaviate o Milvus. Esta integración reduce la complejidad de la arquitectura del sistema a un único gestor de datos que sirve tanto las consultas relacionales del catálogo como las consultas vectoriales del sistema de recomendación.

Se descartó la opción de utilizar un gestor vectorial especializado por la complejidad operativa que introduce: un segundo servicio que desplegar, mantener, monitorizar y sincronizar con la base de datos relacional. Para un TFG con un catálogo de decenas de miles de películas (no millones), el rendimiento de pgvector con un índice HNSW es más que suficiente, como se demuestra en la subsección 7.3.4.

**Esquema normalizado frente a desnormalizado.** El esquema sigue la tercera forma normal (3NF) con tablas de unión para las relaciones muchos-a-muchos (película-género, película-compañía, etc.). Se descartó un esquema desnormalizado (una única tabla ancha con arrays de géneros, JSON de créditos, etc.) por dos motivos: primero, las consultas de filtrado por género, por compañía o por persona son significativamente más eficientes cuando operan sobre tablas normalizadas con índices; segundo, la normalización refleja la estructura natural del dominio y produce un esquema auto-documentado que un evaluador puede comprender inspeccionando únicamente las tablas y sus relaciones.

Para el endpoint de detalle (`GET /movies/{id}`), donde se necesita toda la información agregada en una sola respuesta, se creó una **vista materializable** (`v_movie_details`) que une todas las tablas mediante subconsultas correlacionadas y funciones de agregación (`ARRAY_AGG`, `JSON_AGG`). De este modo, el backend ejecuta una única consulta en lugar de una por cada relación, sin sacrificar la normalización del esquema subyacente.

**Sistema de migraciones propio frente a herramientas existentes.** Se optó por implementar un sistema de migraciones minimalista en Python (sesenta y dos líneas de lógica, excluyendo imports y logging) en lugar de utilizar herramientas establecidas como Flyway o Liquibase. La decisión responde a tres consideraciones: primero, las herramientas existentes están diseñadas para proyectos Java/Kotlin y su integración con el pipeline Python añadiría una dependencia de runtime innecesaria; segundo, las funcionalidades avanzadas que ofrecen (migraciones reversibles, dependencias entre scripts, generación de diferencias automáticas) no se necesitan en un proyecto con un único desarrollador y un esquema que crece de forma lineal; tercero, la implementación propia permite incluir en la memoria una explicación completa y honesta de cómo funciona el sistema, sin delegar la comprensión a una herramienta de terceros.

### 7.3.2.3 Ejecución

**Infraestructura de contenedores (`docker-compose.yml`).** Se configuró un servicio `postgres` basado en la imagen `pgvector/pgvector:pg16`, que incluye PostgreSQL 16 con la extensión pgvector preinstalada. El contenedor expone el puerto 5432, monta un volumen persistente `postgres_data` para los datos, e incluye un *healthcheck* que verifica la disponibilidad del servicio mediante `pg_isready`. Las credenciales se leen de la variable de entorno `DB_PASSWORD` definida en el fichero `.env`, con un valor por defecto para simplificar la primera ejecución.

**Migraciones del esquema.** Se implementaron trece ficheros SQL en el directorio `migrations/`, numerados con la convención `YYYYMMDDNN` (fecha más número secuencial) que garantiza el orden lexicográfico correcto. Las cinco primeras migraciones definen el esquema base:

La migración `2026052500_init_extensions.sql` activa las extensiones `pgvector` (para el tipo `vector` y los operadores de distancia) y `unaccent` (para búsqueda de texto sin sensibilidad a acentos), creando además una función `immutable_unaccent` que permite su uso en índices funcionales (la función `unaccent` estándar de PostgreSQL está marcada como `STABLE`, no como `IMMUTABLE`, lo que impide su uso directo en expresiones de índice).

La migración `2026052501_create_movies_and_collections.sql` crea la tabla `collections` (franquicias como "Toy Story Collection") y la tabla central `movies` con veinticuatro columnas que replican fielmente los campos del CSV procesado: identificadores, metadatos textuales (`title`, `original_title`, `overview`, `tagline`), atributos numéricos (`budget`, `revenue`, `runtime`, `popularity`, `vote_average`, `vote_count`), fecha de estreno, estado (`Released`, `Post Production`, etc.), indicador de contenido adulto, rutas de póster, y una clave foránea a `collections`.

La migración `2026052502_create_people_and_credits.sql` crea la tabla `people` (actores, directores, guionistas) y las tablas de unión `movie_cast` (con campos `character` y `order` para el personaje interpretado y el orden de aparición en créditos) y `movie_crew` (con campos `job` y `department`).

La migración `2026052503_create_categories.sql` crea cinco tablas maestras (`genres`, `production_companies`, `production_countries`, `spoken_languages`, `keywords`) y sus correspondientes cinco tablas de unión con `movies`. Todas las tablas maestras tienen estructura idéntica (`id`, `name`) y todas las tablas de unión tienen estructura idéntica (`movie_id`, `entity_id`, clave primaria compuesta).

La migración `2026052504_create_external_ids.sql` crea la tabla `external_ids` que mapea cada película a sus identificadores en MovieLens (`movielens_id`) e IMDb (`imdb_id`).

Migraciones posteriores (documentadas en las subsecciones 7.3.3 y 7.3.4) añaden la tabla de embeddings, la vista de detalle, los índices de listado y búsqueda, y la evolución del esquema de embeddings.

**Sistema de migraciones (`07_migrate.py`).** El script conecta a PostgreSQL, crea la tabla `schema_migrations` si no existe (con columnas `filename`, `applied_at` y `duration_ms`), lee todos los ficheros `.sql` del directorio `migrations/`, compara sus nombres con los ya registrados en `schema_migrations`, y aplica los pendientes en orden alfabético. Cada migración se ejecuta dentro de una transacción; si falla, la transacción se revierte y el script aborta, dejando la base de datos en un estado consistente. Tras cada migración exitosa, se registra el nombre del fichero y la duración en milisegundos. La idempotencia está garantizada: ejecutar el script dos veces consecutivas no tiene efecto (la segunda ejecución no encuentra migraciones pendientes).

**Scripts de carga de datos.** Los cuatro scripts de carga (`08_load_movies.py`, `09_load_categories.py`, `10_load_credits.py`, `11_load_links.py`) siguen un patrón común: leer el CSV procesado con pandas, parsear los campos JSON embebidos cuando es necesario, acumular las entidades únicas y las filas de unión, e insertarlas en la base de datos. Las entidades maestras (colecciones, géneros, compañías, personas) se insertan con `ON CONFLICT DO NOTHING` para garantizar la idempotencia. Las filas de unión y las películas se cargan mediante la función `COPY` de PostgreSQL (expuesta como `cursor.copy` en psycopg3), que es entre diez y cien veces más rápida que inserts individuales para volúmenes de miles de filas.

El script de categorías (`09_load_categories.py`) es el más complejo de los cuatro: procesa cinco familias de categorías (géneros, compañías productoras, países de producción, idiomas hablados y palabras clave) con un patrón uniforme pero datos de origen diversos (cuatro familias provienen de columnas JSON en `movies.csv`, la quinta proviene de `keywords.csv`).

El orquestador (`12_run_load_pipeline.py`) ejecuta los cuatro scripts en secuencia con el mismo mecanismo que el orquestador de limpieza: ejecución con `subprocess.run`, temporización de cada paso, aborto inmediato si alguno falla. La carga completa del dataset (cuarenta y cinco mil películas, más de un millón de filas de unión, trescientas mil personas) se completa en aproximadamente quince segundos.

**Vistas e índices.** Tras la carga de datos, tres migraciones adicionales optimizan las consultas del backend. La vista `v_movie_details` agrega toda la información de una película en una sola fila, incluyendo géneros, países, idiomas, compañías productoras, palabras clave, reparto y equipo técnico como arrays o JSON, eliminando la necesidad de múltiples joins en el código del backend. Los índices de listado (`2026052601`) cubren las columnas de ordenación y filtrado más frecuentes (`popularity`, `vote_average`, `release_date`, `title`, `original_language`). Los índices de búsqueda (`2026052602`) son funcionales, aplicando la misma normalización que las consultas de búsqueda del backend (`lower`, `unaccent`, reemplazo de guiones por espacios) para que el planificador de PostgreSQL pueda utilizarlos.

### 7.3.2.4 Dificultades encontradas

**Parseo de campos JSON embebidos en CSV.** Los archivos `movies_metadata.csv` y `credits.csv` contienen columnas cuyo valor es un string que representa una lista JSON (por ejemplo, la columna `genres` contiene valores como `[{"id": 28, "name": "Action"}, {"id": 12, "name": "Adventure"}]`). Sin embargo, estos strings no están codificados como JSON estándar: utilizan comillas simples en lugar de dobles, y los valores booleanos aparecen como `True`/`False` (notación Python) en lugar de `true`/`false` (notación JSON). El parseo con `json.loads()` falla en estos casos. La solución implementada utiliza `ast.literal_eval()` de la biblioteca estándar de Python, que interpreta correctamente la notación de literales Python, envuelto en un manejador de errores que devuelve una lista vacía para valores no parseables. Esta función (`parse_python_literal` en el módulo `loading.py`) se reutiliza en todos los scripts de carga que procesan campos JSON.

**Función `unaccent` no utilizable en índices.** PostgreSQL clasifica la función `unaccent` como `STABLE` (su resultado puede variar entre transacciones), lo que impide su uso en expresiones de índice (que requieren funciones `IMMUTABLE`). Dado que en la práctica la función `unaccent` es determinista (su resultado para una entrada dada no cambia nunca), se creó un wrapper `immutable_unaccent` declarado como `IMMUTABLE` que simplemente invoca `unaccent` internamente. Esta técnica es una práctica documentada en la comunidad de PostgreSQL para resolver esta limitación. El wrapper se crea en la primera migración para estar disponible en las migraciones posteriores de índices de búsqueda.

**Orden de carga y claves foráneas.** La tabla `movies` referencia a `collections` mediante una clave foránea. Esto obliga a cargar las colecciones antes que las películas, lo que a su vez obliga a extraer las colecciones del mismo archivo CSV que contiene las películas (`movies.csv`), dado que no existe un archivo separado de colecciones en el dataset original. El script `08_load_movies.py` resuelve esto con un primer pase que extrae y deduplica las colecciones (parseando el campo JSON `belongs_to_collection`), las inserta en la tabla `collections`, y un segundo pase que carga las películas con la referencia ya establecida. El mismo patrón se repite a menor escala en los scripts de categorías (las entidades maestras deben existir antes de que las tablas de unión puedan referenciarlas).
