> **Borrador para revisión.** Tercera subsección del apartado 7.3 ("Ejecutar"), tras 7.3.1 (Limpieza de datos) y 7.3.2 (Base de datos). Cubre la construcción del servidor backend que expone los datos cargados en PostgreSQL como API HTTP.

## 7.3.3 Desarrollo del backend

> *La subsección anterior dejó la base de datos PostgreSQL operativa y poblada con 1.918.063 registros distribuidos en diecisiete tablas relacionales. La función del backend desarrollado en esta subsección es exponer dicha información al frontend mediante una API HTTP bien definida, aislando al cliente de los detalles de almacenamiento y aplicando las reglas de negocio que correspondan.*

### 7.3.3.1 Planificación

El desarrollo del backend constituye el bloque B5 de la planificación temporal expuesta en el apartado 7.1. Conforme a la metodología iterativa e incremental adoptada para el proyecto, este bloque se subdivide en seis sub-pasos de tamaño manejable, cada uno con un incremento funcional verificable al cierre. Esta granularidad busca dos objetivos: por un lado, permitir validar de forma temprana que la base tecnológica elegida (Kotlin sobre la máquina virtual de Java, framework Ktor, conexión a PostgreSQL mediante JDBC con HikariCP) resulta operativa antes de invertir tiempo en lógica de negocio; por otro, facilitar la trazabilidad entre cada commit del repositorio y un objetivo concreto del bloque.

Los seis sub-pasos planificados son los siguientes.

**Sub-paso B5.1 — Configuración inicial y endpoint de salud.** Arrancar el servidor Ktor mínimo con un único endpoint `GET /health` que devuelva respuesta correcta. Su finalidad es verificar la compatibilidad entre las versiones del compilador de Kotlin, el plugin de Gradle, el framework Ktor y el toolchain de Java, así como el funcionamiento del ciclo de arranque/parada del servidor desde el entorno integrado de desarrollo.

**Sub-paso B5.2 — Acceso a la base de datos.** Configurar un pool de conexiones HikariCP que el servidor utilice para alcanzar la base de datos PostgreSQL desplegada en el bloque B4. Ampliar el endpoint de salud para que reporte el estado real del pool, devolviendo un código HTTP distinto si la base de datos no se encuentra disponible. Este sub-paso sirve además para validar la separación entre configuración estática (definida en ficheros YAML) y configuración sensible (definida en variables de entorno).

**Sub-paso B5.3 — Endpoint de detalle de película.** Implementar `GET /movies/{id}`, que devuelve toda la información asociada a una película (datos propios, géneros, países, idiomas, productoras, palabras clave, reparto y equipo técnico) en una única respuesta JSON. Este sub-paso introduce la arquitectura en tres capas (rutas → servicios → repositorios) que vertebra el resto del backend, así como el módulo `:shared` destinado a contener los objetos de transferencia compartibles entre backend y frontend.

**Sub-paso B5.4 — Endpoint de listado paginado.** Implementar `GET /movies` con soporte de filtros combinables (género, año o rango de años, idioma, número mínimo de votos), ordenación configurable por varios criterios y paginación. Este sub-paso introduce la construcción dinámica de consultas SQL con prepared statements y los índices especializados que permiten que las consultas se mantengan rápidas sobre las 45.408 películas del catálogo.

**Sub-paso B5.5 — Endpoint de búsqueda por texto.** Implementar `GET /movies/search`, que devuelve películas cuyo título contenga el texto buscado. La búsqueda debe ser insensible a mayúsculas, a diacríticos y a separadores (espacio frente a guion), de manera que el usuario pueda escribir `amelie`, `spider man` o `GODFATHER` y obtener resultados naturales. Se introduce además un sistema de puntuación de relevancia que ordena los resultados por proximidad al término buscado.

**Sub-paso B5.6 — Suite de pruebas funcionales.** Cubrir los cuatro endpoints anteriores con un conjunto de pruebas de integración que ejerciten tanto los casos felices como los casos de error esperados (entradas inválidas, recursos inexistentes, parámetros fuera de rango). Esta suite debe ejecutarse en pocos segundos, formar parte del repositorio versionado, y servir como red de seguridad ante modificaciones futuras del código.

Cada sub-paso se cierra con un commit en el repositorio Git con mensaje descriptivo, antes de proceder al siguiente. Esta política permite reconstruir el orden cronológico del desarrollo a partir del historial de Git, lo que constituye una fuente complementaria de evidencia para la memoria.

Más allá de la descomposición en sub-pasos, la planificación del bloque incluye tres compromisos arquitectónicos que se respetan a lo largo de toda su ejecución y que conviene anticipar aquí.

**Primero, arquitectura en tres capas estrictamente separadas.** Cada endpoint se implementa como una composición de tres componentes: una *ruta* que recibe la petición HTTP, valida los parámetros sintácticamente y traduce los errores a códigos de estado; un *servicio* que aplica las reglas de negocio (validaciones semánticas, valores por defecto, límites operativos); y un *repositorio* que ejecuta el acceso a la base de datos mediante JDBC. Ninguna ruta accede directamente al repositorio; ningún repositorio decide políticas de negocio. Esta separación facilita el testeo independiente de cada capa y mejora la legibilidad del código.

**Segundo, separación entre módulo de servidor y módulo compartido.** Los objetos de transferencia (DTOs) que viajan en las respuestas JSON viven en un módulo Gradle independiente, denominado `:shared`. Este módulo no contiene lógica, únicamente definiciones de tipos serializables. Su existencia se justifica desde el inicio del bloque pensando en el bloque B7 (frontend), donde el mismo módulo se reutilizará para que cliente y servidor compartan exactamente los mismos tipos. Durante el bloque B5 el módulo `:shared` se mantiene como un proyecto Kotlin de JVM convencional, y su conversión a Kotlin Multiplatform queda diferida al inicio del bloque B7, donde resulta naturalmente necesaria.

**Tercero, contrato de errores estructurado.** Toda respuesta de error producida por el backend devuelve un cuerpo JSON con la forma `{"error": "código_máquina", "message": "mensaje legible"}`. Los códigos máquina permiten al frontend ramificar comportamiento por tipo de error sin tener que parsear cadenas de texto, mientras que el mensaje legible facilita la depuración y, eventualmente, la presentación de mensajes al usuario final. Cada situación de error se asocia a un código distinto: `movie_not_found`, `invalid_id`, `invalid_page_size`, `invalid_year_range`, `invalid_search_query`, `missing_query`, `invalid_query_param`, etcétera.

La estrategia de priorización por riesgo aplicada a la planificación temporal global del proyecto se manifiesta también dentro de este bloque: los sub-pasos B5.1 y B5.2 atacan en primer lugar los aspectos técnicamente más inciertos (compatibilidad de versiones del ecosistema Kotlin, configuración del pool de conexiones), antes de invertir esfuerzo en la lógica de negocio propiamente dicha. Como se documenta en el sub-apartado 7.3.3.4, esta priorización se reveló acertada: el sub-paso B5.1 requirió varias iteraciones de ajuste de versiones antes de quedar operativo, mientras que los sub-pasos posteriores progresaron sin sobresaltos significativos.

### 7.3.3.2 Investigación y elección del stack tecnológico

La elección del lenguaje y del framework para el backend está condicionada por una decisión arquitectónica tomada al inicio del proyecto y documentada en el apartado 6 (Alcance) y en la sección de Tecnologías del apartado 7: el frontend del proyecto se desarrolla en Kotlin sobre Compose Multiplatform compilado a JavaScript. Esta decisión, motivada por la posibilidad de compartir código entre cliente y servidor mediante un módulo común, restringe la elección del backend a tecnologías que puedan cooperar con un módulo Kotlin compartido. La opción natural es desarrollar también el servidor en Kotlin, lo que garantiza coherencia tecnológica y permite reutilizar definiciones de tipos sin intermediarios.

Dentro del ecosistema Kotlin de servidor, las dos opciones principales son **Spring Boot** y **Ktor**. Spring Boot es el framework dominante en el ecosistema Java y dispone de soporte oficial para Kotlin, pero su modelo de configuración basado en convenciones implícitas, autoconfiguración por anotaciones y un ecosistema masivo de starters resulta excesivo para el alcance acotado del presente proyecto. Ktor, mantenido por JetBrains (la misma empresa que desarrolla Kotlin), adopta un modelo radicalmente distinto: la configuración del servidor es explícita y se construye mediante composición de plugins en código, sin descubrimiento automático ni anotaciones. Esto produce arranques más rápidos, una huella de memoria menor y, sobre todo, código en el que es trivial identificar qué funcionalidades están activas. Para un proyecto de tamaño TFG, la simplicidad y transparencia de Ktor priman sobre la madurez ecosistémica de Spring Boot.

La selección de versiones concretas dentro del ecosistema Kotlin requirió una iteración no trivial, documentada en el sub-apartado 7.3.3.4. La combinación finalmente operativa fue Kotlin **2.3.0**, Ktor **3.4.3**, plugin Gradle de Kotlin alineado con el compilador, y un toolchain de Java **21** suministrado a través del Foojay Toolchains Plugin **1.0.0**. Esta combinación, aunque coherente *a posteriori*, no se documenta de forma directa en una fuente canónica única: la tabla de compatibilidad oficial de Ktor no siempre lista las versiones más recientes del compilador de Kotlin, y los archivos `build.gradle.kts` generados por el wizard de IntelliJ pueden quedar desactualizados con respecto a los lanzamientos más recientes de los componentes implicados.

El acceso a la base de datos se resuelve mediante el conector **JDBC oficial de PostgreSQL** (`postgresql:42.7.4`) gestionado por un pool de conexiones **HikariCP** (`com.zaxxer:HikariCP:6.2.1`). Sobre esta combinación se descarta deliberadamente el uso de un ORM (Object-Relational Mapper) como Exposed (de JetBrains) o Hibernate. Esta decisión merece justificación explícita, pues podría parecer una elección regresiva en un contexto donde los ORMs son práctica habitual.

Tres razones la motivan. **Primera**, las consultas que el backend necesita ejecutar son fundamentalmente de **lectura**, con filtros dinámicos y agregaciones a través de varias tablas relacionales. Este patrón se expresa de forma más clara y eficiente en SQL directo que mediante el lenguaje de queries de un ORM, que suele convertirse en una abstracción que oculta más de lo que aclara cuando la consulta es compleja. **Segunda**, la vista de detalle de una película (`GET /movies/{id}`) requiere agregar datos de nueve tablas relacionales en una única respuesta JSON, lo que se resuelve elegantemente mediante una vista SQL con funciones de agregación (`ARRAY_AGG`, `JSON_AGG`) que un ORM no genera de forma natural. **Tercera**, JDBC y SQL son competencias del propio currículo del ciclo formativo de DAM, mientras que el aprendizaje de un ORM concreto reduciría el tiempo dedicado a la lógica del proyecto sin aportar valor académico equivalente.

La serialización de respuestas y peticiones se realiza mediante **kotlinx.serialization** con su backend JSON, biblioteca oficial mantenida por JetBrains. Frente a alternativas como Jackson o Gson, kotlinx.serialization presenta dos ventajas para este proyecto: opera en tiempo de compilación mediante generación de código, lo que evita el uso de reflexión en tiempo de ejecución y produce mejor rendimiento; y está diseñada desde el principio con soporte multiplataforma, lo que permite que las mismas clases serializables del módulo `:shared` se utilicen tanto en el servidor (JVM) como en el frontend (JavaScript) sin adaptaciones.

Para la suite de pruebas funcionales se selecciona el componente **`ktor-server-test-host`**, que constituye la solución oficial para testar aplicaciones Ktor. Su característica diferencial es que no abre un puerto TCP real durante la ejecución de los tests: el cliente HTTP del test se comunica con el servidor en memoria a través de un canal interno, lo que reduce el tiempo de ejecución de cada prueba de varios segundos (los necesarios para arrancar un servidor real) a milisegundos. Combinado con JUnit 5 como framework de aserciones y con la pequeña biblioteca `kotlin-test` que aporta funciones idiomáticas en Kotlin, esto permite ejecutar las veintitrés pruebas funcionales del bloque en menos de dos segundos.

Por último, en lo que respecta a la organización del proyecto en módulos Gradle, se adopta una estructura multi-módulo con tres componentes: un módulo `:server` que contiene la aplicación Ktor; un módulo `:shared` que contiene los tipos serializables compartibles; y un módulo `:composeApp` reservado para el frontend del bloque B7, pendiente al cierre del bloque B5. Esta estructura permite que cada módulo declare sus propias dependencias sin contaminar al resto y prepara el terreno para la conversión del módulo `:shared` a Kotlin Multiplatform cuando llegue el momento.

### 7.3.3.3 Ejecución

Esta sub-sección documenta la ejecución concreta de los seis sub-pasos planificados, en el orden en que se desarrollaron. Para cada sub-paso se describe el incremento funcional aportado al servidor, las decisiones técnicas tomadas durante la implementación y el resultado verificable obtenido al cierre.

#### a) Configuración inicial y endpoint de salud

El primer incremento consiste en poner en marcha un servidor Ktor mínimo con un único endpoint `GET /health` que devuelva una respuesta correcta. Su valor no es funcional sino infraestructural: validar que el stack tecnológico elegido se comporta como se espera, y dejar establecida la estructura del proyecto sobre la que se construirán los siguientes sub-pasos.

La estructura del módulo `:server` se organiza siguiendo el convenio Maven/Gradle estándar: el código fuente vive bajo `src/main/kotlin/com/tfg/movies/server/`, los recursos de configuración bajo `src/main/resources/`, y los tests bajo `src/test/kotlin/`. La función `main` del servidor se reduce a una sola línea (`EngineMain.main(args)`), delegando la configuración completa a un archivo `application.yaml` declarativo. Este archivo define el puerto de escucha, la lista de módulos de aplicación a cargar y los parámetros de base de datos.

Cada módulo de aplicación, en la terminología de Ktor, es una función de extensión sobre la clase `Application` que instala plugins y registra rutas. El proyecto define un único módulo principal, denominado `Application.module()`, que instala los tres plugins necesarios: **ContentNegotiation** con backend JSON para serializar respuestas automáticamente, **CallLogging** para registrar cada petición HTTP en el log, y **CORS** configurado en modo permisivo para desarrollo (a restringir en despliegue real). Este módulo registra además el punto de entrada de cada familia de rutas mediante funciones de extensión sobre `Route`.

El endpoint `GET /health`, en esta fase inicial, devuelve simplemente la cadena `"OK"` con código HTTP 200. La función completa que lo implementa se reduce a tres líneas y serviría únicamente como ping superficial. La verdadera comprobación de salud se introduce en el sub-paso siguiente, una vez establecida la conexión a la base de datos.

El cierre de este sub-paso requirió varias iteraciones de ajuste de versiones, documentadas en detalle en el sub-apartado 7.3.3.4. La combinación final operativa fue Kotlin 2.3.0, Ktor 3.4.3, plugin Foojay 1.0.0 y toolchain de Java 21. Una vez estabilizada, el arranque del servidor desde IntelliJ tarda aproximadamente doscientos milisegundos, y la petición `curl http://localhost:8080/health` devuelve respuesta inmediata.

#### b) Acceso a la base de datos con pool de conexiones

El segundo incremento añade conectividad real a la base de datos PostgreSQL desplegada en el bloque B4, así como un endpoint de salud que reporta el estado efectivo del pool de conexiones.

La conexión se gestiona mediante **HikariCP**, biblioteca de referencia para pools de conexiones JDBC en el ecosistema Java. El motivo de su uso es operativo: abrir una conexión TCP nueva contra PostgreSQL para cada petición HTTP es prohibitivamente lento (decenas de milisegundos por conexión) y desperdicia recursos en ambos extremos. El pool mantiene un conjunto de conexiones abiertas y las reparte entre las peticiones entrantes, devolviéndolas al pool cuando terminan.

La configuración del pool se centraliza en una clase singleton `DatabaseFactory`, expuesta en el paquete `com.tfg.movies.server.config`. Esta clase ofrece tres métodos públicos: `init(config)` arranca el pool leyendo los parámetros de configuración, `close()` lo cierra ordenadamente al apagar el servidor, e `isHealthy()` realiza una consulta trivial (`SELECT 1`) sobre una conexión del pool para verificar que el extremo remoto responde. Los parámetros del pool se definen explícitamente: tamaño máximo configurable por entorno (diez conexiones por defecto), mínimo de dos conexiones inactivas, consulta de verificación al adquirir conexión, tiempo máximo de espera de treinta segundos, tiempo de inactividad de diez minutos antes de cerrar conexiones excedentes, y tiempo máximo de vida de treinta minutos para reciclar conexiones envejecidas.

La configuración sensible (cadena de conexión JDBC, usuario, contraseña) se proporciona al servidor mediante variables de entorno, no mediante valores embebidos en el repositorio. El archivo `application.yaml` referencia dichas variables con sintaxis `$DATABASE_JDBC_URL` y similares, que Ktor resuelve en tiempo de arranque contra el entorno del proceso. Durante el desarrollo, estas variables se definen mediante una configuración de ejecución en IntelliJ que las inyecta al lanzar la aplicación; en despliegue real provendrían del orquestador correspondiente.

Una decisión específica de esta fase merece comentario. El driver JDBC de PostgreSQL requiere una cadena de conexión con sintaxis ligeramente distinta a la que utiliza la biblioteca libpq de Python: aquélla espera el prefijo `jdbc:postgresql://`, mientras que ésta se conforma con `postgresql://`. Para evitar duplicar la configuración o forzar al desarrollador a recordar dos formatos, se introdujeron tres variables de entorno separadas: `DATABASE_JDBC_URL` para la conexión JDBC, junto con `DATABASE_USER` y `DATABASE_PASSWORD` para las credenciales. La variable `DATABASE_URL` original (formato libpq) se mantiene para uso del pipeline de carga de datos en Python.

El endpoint `GET /health` se amplía en este sub-paso para devolver un cuerpo JSON con dos campos: `status` y `database`. Cuando la base de datos responde correctamente, el endpoint devuelve código HTTP 200 con `{"status": "ok", "database": "up"}`. Cuando la base de datos no está accesible (contenedor parado, credenciales incorrectas, red caída), devuelve código HTTP 503 (Service Unavailable) con `{"status": "degraded", "database": "down"}`. Este comportamiento se verifica empíricamente parando el contenedor de PostgreSQL, observando el cambio de respuesta, y volviendo a arrancarlo, observando la recuperación automática del pool.

#### c) Endpoint de detalle de película

El tercer incremento implementa `GET /movies/{id}`, primer endpoint con valor funcional para el frontend. Devuelve toda la información asociada a una película en una única respuesta JSON: campos propios (título, sinopsis, fecha de estreno, duración, presupuesto, recaudación, valoración media, etcétera), géneros, países de producción, idiomas hablados, productoras, palabras clave, reparto completo con personajes interpretados y orden de aparición, y equipo técnico con departamentos y oficios.

Este sub-paso introduce dos decisiones arquitectónicas que vertebran el resto del backend.

**La arquitectura en tres capas.** La gestión del endpoint se reparte entre tres clases con responsabilidades claramente separadas. La clase `MovieRoutes` gestiona el aspecto HTTP: lee el parámetro de ruta, lo valida sintácticamente (debe ser un entero), invoca al servicio y traduce el resultado a la respuesta correspondiente. La clase `MovieService` aplica reglas de negocio (en este sub-paso, mínimas: comprobar que el identificador es positivo); se introduce desde el principio para que la posición esté preparada cuando lleguen sub-pasos con reglas más sustanciales. La clase `MovieRepository` ejecuta la consulta SQL contra la base de datos y mapea el `ResultSet` resultante al objeto de transferencia. Esta separación facilita el testeo de cada capa de forma independiente y mejora la mantenibilidad: si en el futuro cambia el motor de base de datos, sólo el repositorio se ve afectado; si cambian las reglas de negocio, sólo el servicio.

**La vista SQL `v_movie_details`.** Para construir la respuesta de detalle resulta necesario combinar información de nueve tablas relacionales (movies, collections, genres, production\_countries, spoken\_languages, production\_companies, keywords, movie\_cast con people, movie\_crew con people). Implementar esta combinación como una serie de `JOIN` directos genera una explosión cartesiana: si una película tiene tres géneros, dos países y setenta y siete actores, el `JOIN` ingenuo produce trescientas sesenta y dos filas que después hay que deduplicar en código. La solución adoptada es definir una vista SQL llamada `v_movie_details` que utiliza subconsultas con funciones de agregación (`ARRAY_AGG` para listas de cadenas, `JSON_AGG` para listas de objetos con varios campos). La vista devuelve exactamente una fila por película, donde las relaciones aparecen como arrays o como cadenas JSON ya construidas por PostgreSQL. El repositorio consulta esta vista mediante una única consulta parametrizada, lo que elimina por completo la lógica de agregación del lado de la aplicación.

Los objetos de transferencia se definen como **data classes** con anotación `@Serializable` y residen en el módulo `:shared`. La clase principal `MovieDetails` se compone de tipos primitivos y de listas de otras data classes específicas: `CastMember`, `CrewMember`, `CollectionRef`, `CrossReferences`. Esta descomposición refleja directamente la estructura del JSON devuelto al cliente y proporciona seguridad de tipos en ambos extremos.

Como decisión de alcance, dos campos se omiten deliberadamente del objeto de respuesta. La columna `homepage` se descarta por su baja cobertura en el dataset original (menos del cinco por ciento de las películas la tienen rellena) y por ser información de utilidad marginal. La columna `backdrop_path` (imagen panorámica de fondo), pese a su utilidad indudable para la interfaz, no existe en el dataset Banik utilizado como fuente; su incorporación queda documentada como ampliación futura mediante consultas a la API oficial de TMDB.

El contrato de errores del endpoint distingue tres situaciones. La petición de un identificador existente devuelve código 200 con el JSON completo. La petición de un identificador inexistente devuelve código 404 con cuerpo `{"error": "movie_not_found", "message": "..."}`. La petición con un identificador no numérico (por ejemplo, `/movies/abc`) devuelve código 400 con cuerpo `{"error": "invalid_id", "message": "..."}`. Esta diferenciación entre "el cliente formuló mal la petición" (400) y "la petición es válida pero el recurso no existe" (404) se mantiene a lo largo de todos los endpoints del backend.

La verificación empírica del sub-paso se realiza con la película *Fight Club* (identificador TMDB 550), elegida por tener un volumen significativo de relaciones (setenta y siete miembros del reparto, ciento siete miembros del equipo técnico, varios géneros y países, palabras clave abundantes). La respuesta JSON resultante ocupa aproximadamente nueve kilobytes y se genera en menos de veinte milisegundos.

#### d) Endpoint de listado paginado

El cuarto incremento implementa `GET /movies`, endpoint que devuelve una página de películas que cumplen un conjunto opcional de filtros y siguen un criterio configurable de ordenación. Este sub-paso es el más rico técnicamente del bloque: introduce construcción dinámica de consultas SQL, gestión correcta de paginación sobre datasets grandes y un sistema de filtros componibles que sirve de plantilla para futuros endpoints similares.

El contrato del endpoint admite los siguientes parámetros de consulta, todos opcionales. Los filtros propiamente dichos son `genre` (admitiendo múltiples valores con semántica de unión), `year` (año exacto de estreno), `yearFrom` y `yearTo` (rango de años, ambos inclusivos), `language` (código ISO 639-1) y `minVoteCount` (umbral mínimo de votos como filtro de calidad). La ordenación se controla con `sortBy` (uno entre `popularity`, `voteAverage`, `voteCount`, `releaseDate`, `title`) y `sortDir` (`asc` o `desc`). La paginación utiliza `page` (entero positivo, partiendo de uno) y `pageSize` (entre uno y cien). Los valores por defecto, cuando un parámetro no se proporciona, son `sortBy=popularity`, `sortDir=desc`, `page=1`, `pageSize=20`, sin filtros aplicados.

La estructura de la respuesta es un objeto JSON con cinco campos: `items` (array de objetos `MovieSummary` correspondientes a la página actual), `page` (página actual), `pageSize` (tamaño solicitado), `total` (número total de películas que cumplen los filtros, sin tener en cuenta la paginación) y `totalPages` (cantidad total de páginas, calculada como `ceil(total / pageSize)`). El campo `total` permite al frontend renderizar controles de paginación correctos sin tener que recorrer todas las páginas.

El objeto `MovieSummary`, definido en el módulo `:shared`, es una versión ligera de `MovieDetails` orientada a listados: contiene únicamente los campos necesarios para una tarjeta de presentación (identificador, título, año, ruta del póster, valoración media, número de votos, popularidad, géneros). Esta separación reduce considerablemente el tamaño de las respuestas: una página de veinte películas en formato `MovieSummary` ocupa aproximadamente cuatro kilobytes, frente a los aproximadamente ciento ochenta que ocuparían si se utilizase `MovieDetails`.

La implementación en el repositorio combina varias técnicas que merecen explicación detallada.

**Construcción dinámica de consultas con prepared statements.** La consulta SQL debe variar según qué filtros estén presentes, pero ningún valor procedente del cliente puede concatenarse directamente al texto SQL sin abrir la puerta a inyección. La solución adoptada es acumular en paralelo dos colecciones: una lista de fragmentos `WHERE` que utilizan marcadores de posición (`?`) y una lista de valores que se asociarán a esos marcadores al ejecutar el `PreparedStatement`. Una clase interna `WhereBuilder` encapsula esta acumulación y produce al final una cláusula `WHERE` correctamente formada junto con la lista ordenada de parámetros. De este modo, ningún valor del cliente acaba en el texto SQL como cadena concatenada; todos viajan como parámetros enlazados que el driver JDBC escapa por construcción.

**Filtro por género con `EXISTS` en lugar de `JOIN` + `DISTINCT`.** La relación entre películas y géneros es de muchos a muchos, gestionada por la tabla puente `movie_genres`. Para devolver las películas que pertenezcan a un determinado género existen dos enfoques en SQL: un `JOIN` contra `movie_genres` seguido de un `DISTINCT` para eliminar duplicados, o una subconsulta `EXISTS` que se limite a comprobar la existencia de al menos una fila coincidente. Se elige el segundo. La motivación es doble. Por un lado, el motor de PostgreSQL puede cortocircuitar la evaluación de un `EXISTS` en cuanto encuentre la primera coincidencia, sin necesidad de generar todas las combinaciones. Por otro, mantener la cardinalidad estable (una sola fila por película en el resultado intermedio) simplifica significativamente el cálculo correcto del `COUNT(*)` requerido para la paginación; con un `JOIN` + `DISTINCT` el conteo se complica y suele dar lugar a errores sutiles.

**Tiebreaker explícito en `ORDER BY` para garantizar paginación estable.** Cuando se ordena por una columna que admite empates (por ejemplo, dos películas con la misma popularidad o el mismo año), PostgreSQL no garantiza un orden determinista entre los registros empatados. Esto produce un problema sutil al paginar: una misma fila puede aparecer en dos páginas distintas en ejecuciones sucesivas de la consulta, o no aparecer en ninguna. La solución consiste en añadir un segundo criterio de ordenación por la clave primaria (`m.id ASC`), que es única por definición. El criterio combinado `(columna_principal, m.id)` es siempre totalmente ordenable, lo que garantiza una paginación reproducible.

**Cláusula `NULLS LAST` en ordenaciones descendentes.** PostgreSQL, por defecto, sitúa los valores `NULL` al principio de las ordenaciones descendentes. Este comportamiento es contraintuitivo en el contexto del listado: si se ordena por valoración media de mayor a menor, no resulta deseable que la primera página esté dominada por películas sin valoración. La cláusula `NULLS LAST` corrige esta ordenación situando los nulos al final.

**Decisión de diseño: valor por defecto inteligente para `minVoteCount`.** Una observación de los datos revela que ordenar por valoración media sin ningún filtro produce una lista dominada por películas con uno o dos votos perfectos (`vote_average = 10.0`), estadísticamente irrelevantes. Para que la ordenación por valoración produzca resultados útiles por defecto, el servicio aplica automáticamente `minVoteCount = 10` cuando se ordena por `voteAverage` y el cliente no ha especificado un valor propio. Esta política se documenta explícitamente y el cliente puede anularla pasando `minVoteCount=0`. Es un compromiso entre comodidad por defecto y respeto de la intención del usuario cuando éste expresa una distinta.

**Migración SQL con índices especializados.** Para mantener el endpoint operativo sobre las cuarenta y cinco mil películas se añadió una nueva migración (`2026052601_add_listing_indexes.sql`) que crea siete índices: índices descendentes con `NULLS LAST` para las columnas de ordenación frecuente, un índice funcional sobre `EXTRACT(YEAR FROM release_date)::INT` que permite que el filtro por año aproveche un índice en lugar de degradarse a escaneo completo, e índices parciales (`WHERE col IS NOT NULL`) sobre columnas con porcentajes altos de nulos para reducir el tamaño de cada índice.

La validación del endpoint distingue tres categorías de errores con códigos diferenciados. Un parámetro presente pero con valor sintácticamente incorrecto (por ejemplo, `year=abc`) produce código 400 con `error: "invalid_query_param"`. Un parámetro sintácticamente correcto pero fuera de los límites operativos (por ejemplo, `pageSize=500`) produce código 400 con `error: "invalid_page_size"`. Una combinación de parámetros internamente incoherente (por ejemplo, `yearFrom=2020&yearTo=2010`) produce código 400 con `error: "invalid_year_range"`. Esta diferenciación entre errores de análisis sintáctico y errores de validación semántica responde a una práctica habitual en diseño de APIs y permite al frontend ofrecer mensajes específicos al usuario.

#### e) Endpoint de búsqueda por texto

El quinto incremento implementa `GET /movies/search?q=...`, que devuelve películas cuyo título contiene el texto buscado, ordenadas por relevancia. La búsqueda se diseña para resultar natural al usuario hispanohablante: insensible a mayúsculas, insensible a diacríticos e insensible a separadores entre palabras.

Tres enfoques principales se consideraron para implementar la búsqueda por texto en PostgreSQL. **Primero, el operador `ILIKE`** sobre la columna de título, eventualmente combinado con normalización mediante la extensión `unaccent`. **Segundo, búsqueda de texto completo** (`tsvector`/`tsquery`), que ofrece capacidades lingüísticas avanzadas como stemming y eliminación de palabras vacías. **Tercero, la extensión `pg_trgm`**, basada en similitud de trigramas, que aporta tolerancia a errores tipográficos.

Se elige el primero. La búsqueda por texto completo se descarta porque el stemming depende del idioma configurado, y el catálogo del proyecto incluye títulos en más de setenta idiomas distintos sin posibilidad de identificarlos *a priori* en el momento de la búsqueda. La búsqueda por trigramas se descarta porque la tolerancia a errores produce ordenaciones impredecibles para usuarios que escriben títulos exactos: una consulta por "matrix" podría devolver "Mtrix" antes que "The Matrix" si el primer título tuviera mayor similitud trigrámica con la consulta. El operador `ILIKE`, en cambio, ofrece coincidencias exactas predecibles, reutiliza el índice funcional ya disponible de la base de datos y resulta suficiente para un catálogo donde el caso de uso principal es localizar películas conocidas por su título.

La búsqueda se realiza sobre dos columnas: `title` y `original_title`. Esta segunda columna añade valor cuando los títulos difieren entre versiones (por ejemplo, una película japonesa cuyo `title` está en inglés y cuyo `original_title` está en japonés). Se descarta extender la búsqueda al campo `overview` porque los resultados resultarían poco relevantes: las películas que mencionan un término en su sinopsis no son normalmente las que el usuario quiere encontrar cuando escribe ese término.

La normalización del texto se aplica de forma consistente en ambos extremos de la comparación: tanto sobre la columna almacenada como sobre el término de búsqueda. La expresión SQL completa es `replace(lower(immutable_unaccent(col)), '-', ' ')`, que pasa el texto a minúsculas, elimina los diacríticos mediante la función `immutable_unaccent` (variante determinista de `unaccent` requerida para indexación funcional) y sustituye los guiones por espacios. Esta última normalización permite que un usuario escriba `spider man` y encuentre `Spider-Man`, o viceversa, sin necesidad de conocer el formato exacto del título.

Para que la consulta resulte eficiente sobre las cuarenta y cinco mil filas, se introduce una migración adicional (`2026052602_add_search_indexes.sql`) con dos índices funcionales construidos exactamente sobre la misma expresión normalizadora utilizada en la cláusula `WHERE`. La igualdad entre la expresión indexada y la expresión consultada es condición necesaria para que PostgreSQL aplique el índice; cualquier divergencia, por mínima que parezca, fuerza un escaneo completo de la tabla.

**Sistema de puntuación de relevancia.** Para que los resultados respeten una noción intuitiva de relevancia, la consulta calcula una puntuación numérica para cada coincidencia mediante una expresión `CASE WHEN`: cien puntos para coincidencia exacta del título completo, cincuenta puntos cuando el título comienza por el término buscado, diez puntos cuando el título contiene el término en cualquier posición, y cinco puntos cuando la coincidencia se produce en el título original pero no en el título principal. La cláusula `ORDER BY` utiliza esta puntuación como criterio primario, seguida de la popularidad descendente como desempate, y del identificador ascendente como tiebreaker final. Esta combinación garantiza que una búsqueda por "matrix" devuelva *The Matrix* (1999) antes que *The Matrix Reloaded*, aun cuando esta segunda película pudiera tener popularidad ligeramente superior.

La validación del término de búsqueda exige longitud mínima de dos caracteres (un solo carácter produce listas demasiado largas para ser útiles) y máxima de cien (cualquier valor superior es claramente abusivo). Una petición con `q` ausente o vacío produce código 400 con `error: "missing_query"`. Una petición con `q` fuera de los límites de longitud produce código 400 con `error: "invalid_search_query"`. La paginación reutiliza los parámetros `page` y `pageSize` del endpoint de listado.

Una limitación del enfoque, documentada honestamente, es que la búsqueda opera contra los títulos almacenados en la base de datos, que son los que figuran en el dataset Banik. Este dataset contiene la versión "canónica" del título según TMDB (habitualmente en inglés) y el título original (en el idioma de producción), pero no las traducciones a otros idiomas. En consecuencia, una búsqueda por "Padrino" no localiza *The Godfather*. El enriquecimiento con traducciones mediante la API oficial de TMDB queda documentado como ampliación futura.

#### f) Suite de pruebas funcionales

El sexto y último sub-paso del bloque incorpora una suite de pruebas funcionales que verifica el comportamiento de los cuatro endpoints implementados. La suite ejerce el sistema completo (capa de rutas, capa de servicios, capa de repositorios, conexión real a base de datos) ante un conjunto representativo de escenarios, tanto de éxito como de error, y constituye la red de seguridad sobre la que se sostiene cualquier modificación futura del código.

Se opta por **pruebas de integración** sobre el sistema completo, frente a pruebas unitarias aisladas de cada clase. Esta elección responde a la naturaleza concreta de la arquitectura desarrollada: las clases de servicio del backend son fundamentalmente delegadores (validan input y reenvían al repositorio); los repositorios contienen lógica acoplada a SQL difícilmente representable con tipos abstractos; y la mayor parte del valor del sistema reside en que las tres capas cooperen correctamente sobre datos reales. Las pruebas unitarias clásicas requerirían la introducción artificial de mocks o stubs para aislar cada clase, sin aportar garantía adicional sobre el funcionamiento conjunto del sistema. Las pruebas de integración, por el contrario, ejercitan precisamente esa cooperación.

El componente técnico que habilita estas pruebas con bajo coste de ejecución es `ktor-server-test-host`. Su característica clave es que arranca la aplicación Ktor en el mismo proceso de la máquina virtual de Java del test, sin abrir un socket TCP real: el cliente HTTP del test se comunica con el servidor a través de un canal interno. Esto reduce el tiempo de cada prueba a milisegundos, frente a los varios segundos que tardaría arrancar un servidor real, lo que permite ejecutar la suite completa en menos de dos segundos.

Una decisión deliberada de alcance es **reutilizar la base de datos de desarrollo** para los tests en lugar de levantar una instancia aislada mediante Testcontainers. Esta decisión se justifica por dos motivos. Primero, todos los endpoints del bloque B5 son de sólo lectura, por lo que no existe riesgo de mutación o contaminación de datos. Segundo, la base de datos de desarrollo ya está poblada con el dataset completo, lo que permite que las pruebas se ancoren a películas reales y conocidas (por ejemplo, *Fight Club* con identificador 550, *The Matrix* con identificador 603, *Amélie* con identificador 194) cuyas propiedades son estables y verificables. La introducción de Testcontainers queda planificada para el bloque B7, cuando aparezcan endpoints de escritura y la aislación del entorno de pruebas resulte necesaria.

La suite se estructura en cuatro archivos, uno por endpoint testeado, en paquetes paralelos a los del código de producción. La organización facilita la navegación: encontrar las pruebas asociadas a un endpoint requiere abrir el archivo correspondiente, sin búsquedas. Un archivo auxiliar `TestApp.kt` factoriza la lógica común de arranque del servidor de test mediante una función `runWithApp` que recibe el bloque de prueba como parámetro lambda, instala la configuración de aplicación inline (sin depender de variables de entorno), crea un cliente HTTP con negociación de contenido configurada y entrega ambos al test.

El conjunto final consta de veintitrés pruebas distribuidas así: una prueba sobre `/health`, tres sobre `/movies/{id}` (existente, inexistente, identificador no numérico), diez sobre `/movies` (paginación, filtros simples, filtros combinados, ordenación, errores de validación) y nueve sobre `/movies/search` (búsqueda simple, relevancia, normalización por mayúsculas, normalización por diacríticos, normalización por separadores, errores). Las aserciones se formulan sobre propiedades estructurales (códigos de estado, tamaños de página, presencia de identificadores específicos, contenido de listas) en lugar de sobre el contenido exacto del JSON, lo que las hace robustas ante cambios menores en el formato sin perder capacidad de detectar regresiones.

La ejecución completa de la suite, mediante el comando `./gradlew :server:test`, produce un informe HTML en `server/build/reports/tests/test/index.html` con el desglose de cada prueba y su tiempo individual. Al cierre del sub-paso, las veintitrés pruebas pasan correctamente en menos de dos segundos.

### 7.3.3.4 Redacción y dificultades encontradas

A diferencia de los sub-apartados anteriores del bloque, donde el sub-paso planificado coincide con razonable fidelidad con el sub-paso ejecutado, el bloque B5 incluyó tres dificultades técnicas significativas y dos decisiones reconsideradas durante la implementación. Su documentación honesta forma parte del contrato académico del proyecto y de la metodología de "autocrítica documentada" expuesta en el apartado 7.1.

**Iteración inicial sobre versiones del ecosistema Kotlin.** El sub-paso B5.1, planificado como una configuración rutinaria del proyecto, requirió en la práctica seis ciclos de modificación del archivo `build.gradle.kts` antes de alcanzar una combinación operativa. Los problemas se manifestaron con mensajes de error opacos del plugin Gradle de Kotlin que no apuntaban directamente al origen del conflicto. La tabla oficial de compatibilidad de Ktor no incluía las versiones más recientes del compilador de Kotlin que el wizard de IntelliJ había sugerido inicialmente, y la combinación generada automáticamente por dicho wizard resultó inestable. La solución consistió en consultar el repositorio oficial de Maven Central para verificar qué versión de Ktor era efectivamente publicada para qué versión de Kotlin, retroceder a una combinación más conservadora, e incorporar el plugin Foojay Toolchains Resolver en su versión 1.0.0 para garantizar que el toolchain de Java 21 quedase resuelto sin depender de la instalación local. La combinación final operativa (Kotlin 2.3.0, Ktor 3.4.3, Foojay 1.0.0, JDK 21) quedó consolidada al cierre del sub-paso B5.1 y se mantuvo sin modificaciones durante el resto del bloque.

Esta dificultad refuerza la importancia metodológica de la estrategia de priorización por riesgo expuesta en el apartado 7.1: precisamente por sospechar que la elección de versiones podía resultar problemática, este aspecto se abordó en el primer sub-paso del bloque, antes de invertir esfuerzo en lógica de aplicación. De haberse pospuesto, el descubrimiento del problema en mitad del desarrollo del endpoint de detalle habría supuesto la pérdida de trabajo previo y una restructuración mayor.

**Descubrimiento de la limitación con separadores durante la verificación de la búsqueda.** El sub-paso B5.5, dedicado al endpoint de búsqueda, se completó inicialmente con búsqueda insensible a mayúsculas y a diacríticos, considerándose terminado. Durante la batería de verificaciones empíricas posterior, sin embargo, se observó que la consulta `?q=spider+man` (con espacio) devolvía resultado vacío, pese a que el catálogo contiene varias películas de la franquicia Spider-Man. La causa resultó ser que el dataset almacena los títulos con guion (`Spider-Man`) y que el operador `ILIKE` realiza coincidencia literal, sin equiparar guiones y espacios. Este caso, no contemplado en el diseño original, motivó una iteración adicional dentro del sub-paso: ampliación de la expresión de normalización con `replace(..., '-', ' ')`, sustitución del índice funcional preexistente por dos nuevos índices construidos sobre la expresión ampliada (migración `2026052602_add_search_indexes.sql`), y verificación de que las cuatro pruebas afectadas continuaban pasando. La iteración añadió aproximadamente noventa minutos al sub-paso planificado, pero produjo un endpoint significativamente más usable.

**Acceso a configuración como lista en pruebas con `MapApplicationConfig`.** Durante el desarrollo del sub-paso B5.6 se produjo un fallo en tiempo de ejecución que llevó varios intentos resolver: las pruebas de integración no podían arrancar el servidor de test, devolviendo el error `Property ktor.application.modules.size not found`. La causa, no documentada en la guía oficial de Ktor para `MapApplicationConfig`, es que las listas en este componente no se representan con la sintaxis natural de Kotlin sino mediante una convención propia que utiliza claves auxiliares `.size` y `.0`, `.1`, etcétera. La solución, una vez identificada la causa, fue trivial; el coste estuvo en la búsqueda del diagnóstico. Esta convención no aparece en la documentación principal de Ktor y sólo se localiza en código fuente del propio framework.

**Decisión reconsiderada: gestión de parámetros enteros opcionales con dos sobrecargas.** El primer borrador del archivo `MovieRoutes.kt` definió una única función `parseIntQuery` que devolvía siempre un tipo `Int?` (entero anulable). El compilador de Kotlin rechazó las llamadas en las que dicho resultado se asignaba a un parámetro no anulable (como `page` o `pageSize`, donde la presencia del valor por defecto garantiza la ausencia de nulo). La resolución consistió en sustituir la función única por dos sobrecargas siguiendo la convención canónica de la biblioteca estándar de Kotlin: `parseIntQuery(name, default: Int): Int` para parámetros con valor por defecto no nulo, y `parseIntQueryOrNull(name): Int?` para parámetros realmente opcionales. La doble sobrecarga, además de resolver el error de tipos, mejora la legibilidad del código del endpoint al hacer explícita en cada llamada la intención semántica.

**Limitaciones conocidas documentadas como ampliaciones futuras.** Al cierre del bloque B5 persisten dos limitaciones identificadas y conscientemente aceptadas. Primera, el dataset Banik no contiene la ruta de la imagen panorámica de fondo (`backdrop_path`) ni traducciones de los títulos a idiomas distintos del original e inglés. La incorporación de ambos elementos mediante consultas a la API oficial de TMDB queda planificada como ampliación. Segunda, los tests del bloque reutilizan la base de datos de desarrollo; la migración a Testcontainers se realizará al inicio del bloque B7, cuando los endpoints de escritura del frontend hagan necesario el aislamiento del entorno de prueba.

Con estas dificultades documentadas y resueltas, el bloque B5 se cierra con un backend funcional que cumple todos los objetivos planificados: cuatro endpoints HTTP operativos (`/health`, `/movies/{id}`, `/movies`, `/movies/search`), arquitectura en tres capas estrictamente separadas, módulo `:shared` preparado para su conversión a Kotlin Multiplatform al inicio del bloque B7, contrato de errores estructurado y consistente entre todos los endpoints, y suite de veintitrés pruebas funcionales que ejecuta en menos de dos segundos. El backend queda en condiciones de servir como cimentación tanto para el sistema de recomendación basado en embeddings (bloque B6) como para la aplicación cliente desarrollada en Compose Multiplatform (bloque B7).
