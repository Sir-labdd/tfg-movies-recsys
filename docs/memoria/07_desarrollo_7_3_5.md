## 7.3.5 Desarrollo de la aplicación cliente

> *Las subsecciones anteriores dejaron un backend completo: cuatro endpoints HTTP para listado, detalle, búsqueda y recomendación de películas, respaldados por una base de datos PostgreSQL con más de cuarenta y cuatro mil embeddings semánticos indexados con HNSW. Sin embargo, el sistema no era accesible más que por línea de comandos o por herramientas como cURL. Esta subsección cierra esa brecha construyendo una aplicación cliente web completa en Kotlin, ejecutada íntegramente en el navegador, que consume todos los endpoints del backend y ofrece al usuario una experiencia de exploración visual del catálogo con filtros, búsqueda en tiempo real, detalle de películas y recomendaciones semánticas.*

### 7.3.5.1 Planificación

El desarrollo de la aplicación cliente constituye el bloque B7 de la planificación temporal expuesta en el apartado 7.1. Es el bloque más extenso del proyecto tanto en número de sub-pasos como en variedad técnica: mientras que los bloques anteriores se movían dentro de un mismo dominio tecnológico (Python para limpieza, SQL para base de datos, Kotlin/JVM para el servidor), este bloque introduce un target completamente nuevo (Kotlin/JS compilado a JavaScript), una biblioteca de UI declarativa (Compose HTML), un sistema de empaquetado (Webpack), un cliente HTTP específico para el navegador (Ktor Client JS) y un mecanismo de enrutamiento propio del lado del cliente.

La descomposición en ocho sub-pasos sigue un orden de dependencias técnicas ascendentes: las capas de infraestructura (configuración de Gradle, módulo compartido, cliente HTTP) se construyen primero porque son prerrequisitos de las pantallas; las pantallas se construyen en orden de complejidad creciente (listado antes que detalle, detalle antes que búsqueda); y las funcionalidades transversales (theming, servicio estático) se añaden al final porque operan sobre pantallas ya construidas.

**Sub-paso B7.1 — Centralización de versiones y módulo compartido multiplatforma.** Migrar las versiones de dependencias dispersas en los ficheros `build.gradle.kts` de cada módulo a un **Gradle Version Catalog** (`gradle/libs.versions.toml`), que centraliza las declaraciones en un único fichero y genera *type-safe accessors* para usarlas desde los scripts de build. Adicionalmente, convertir el módulo `:shared` de Kotlin/JVM puro a **Kotlin Multiplatform** con targets `jvm()` y `js(IR) { browser() }`, de modo que los DTOs definidos en `commonMain` (como `MovieSummary`, `PaginatedResponse` y `SimilarMoviesResponse`) puedan compilarse tanto para el servidor como para el cliente web, sin duplicación de código.

**Sub-paso B7.2 — Creación del módulo `:composeApp`.** Crear un nuevo módulo Gradle con los plugins `kotlin("multiplatform")` y `org.jetbrains.compose`, configurado para el target `js(IR) { browser { binaries.executable() } }`. Este módulo alberga todo el código de la aplicación cliente y produce un bundle JavaScript ejecutable en el navegador. La configuración incluye la resolución de dependencias de Compose Multiplatform (runtime, HTML core) y Ktor Client (JS engine, Content Negotiation, kotlinx-serialization-json), así como la integración con Webpack para el empaquetado del bundle de desarrollo y producción.

**Sub-paso B7.3 — Cliente HTTP tipado.** Implementar un paquete `api/` dentro de `:composeApp` que encapsula toda la comunicación con el backend. El paquete expone un objeto `MovieApi` con cuatro métodos suspending (`getMovies`, `getMovie`, `searchMovies`, `getSimilarMovies`) que devuelven `Result<T>`, donde el tipo `T` reutiliza los DTOs del módulo `:shared`. Los errores HTTP se mapean a una excepción tipada `ApiException` con campos de estado, código de error máquina y mensaje legible. La URL base del backend es configurable en tiempo de ejecución mediante `ApiConfig`, que autodetecta si el frontend se ejecuta desde el servidor de desarrollo (puerto 8081, apuntando a `localhost:8080`) o desde el propio Ktor en producción (rutas relativas, mismo origen).

**Sub-paso B7.4 — Pantalla de listado con filtros e infinite scroll.** Construir la pantalla principal del catálogo, que incluye un sistema de diseño basado en CSS *custom properties* (paleta oscura, tipografía de sistema, escala de espaciados de 4 px), un componente `MovieCard` que renderiza pósteres reales de TMDB con fallback visual para imágenes no disponibles, un grid responsive mediante CSS Grid, una barra de filtros (género, ordenación, dirección) cuyos géneros se obtienen dinámicamente de un nuevo endpoint `GET /genres`, paginación infinita por detección de proximidad al final del viewport, y un botón flotante de retorno al inicio. Este sub-paso es el más extenso del bloque y se descompone internamente en cinco fases: tema base, componente de tarjeta, grid estático con datos mock, conexión real al backend, y filtros con infinite scroll.

**Sub-paso B7.5 — Pantalla de detalle y navegación.** Implementar la pantalla de detalle de una película, que muestra toda la información devuelta por el endpoint `GET /movies/{id}` (póster en formato ampliado, título, tagline, año, duración, valoración, sinopsis, reparto) junto con una sección de recomendaciones similares obtenida del endpoint `GET /movies/{id}/similar`. Para llegar a esta pantalla se introduce un mecanismo de navegación del lado del cliente basado en fragmentos de URL (hash routing: `/#/movie/{id}`), gestionado por un objeto `AppState` que integra con la API `History` del navegador para que los botones de avance y retroceso funcionen como el usuario espera.

**Sub-paso B7.6 — Búsqueda con autocompletado y barra de navegación.** Añadir una barra de navegación persistente (*sticky header*) visible en todas las pantallas, con el título de la aplicación como enlace al listado y un campo de búsqueda con autocompletado en tiempo real. El autocompletado dispara búsquedas tras un *debounce* de trescientos milisegundos y muestra hasta ocho sugerencias con miniatura, título, año, valoración y géneros. Pulsar Enter o el botón de búsqueda navega a una pantalla de resultados completa con hasta cincuenta películas.

**Sub-paso B7.7 — Modo claro y modo oscuro.** Definir un segundo conjunto de valores para las CSS *custom properties* bajo una clase `.light-theme`, permitir la alternancia entre temas mediante un botón de sol/luna en la barra de navegación, y persistir la preferencia del usuario en `localStorage` con respeto al ajuste del sistema operativo (`prefers-color-scheme`) como valor por defecto.

**Sub-paso B7.8 — Servir el frontend desde Ktor y configuraciones de ejecución.** Configurar Gradle para que el build de producción del frontend (bundle minificado de aproximadamente 1,19 MiB) se copie al classpath del servidor, de modo que Ktor sirva tanto la API como la aplicación cliente desde un único proceso y un único puerto. Crear cuatro configuraciones de ejecución de IntelliJ compartidas como ficheros de proyecto para estandarizar el flujo de desarrollo.

### 7.3.5.2 Investigación

La investigación técnica previa a la implementación del cliente abordó tres áreas diferenciadas: la elección de la tecnología de interfaz de usuario, el modelo de navegación y el mecanismo de estilado.

**Compose Multiplatform para la Web frente a alternativas.** La decisión principal del bloque fue la elección de **Compose HTML** como tecnología de interfaz. La familia Compose Multiplatform ofrece dos modos de ejecución para la web: Compose HTML (también denominado *Compose for Web DOM*) y Compose for Canvas (anteriormente Compose for Web Canvas, basado en Skia/Skiko). Compose for Canvas dibuja la interfaz sobre un elemento `<canvas>` de HTML5 utilizando el mismo motor de renderizado que Compose para escritorio y Android, lo que permite compartir código visual al cien por cien entre plataformas; sin embargo, el contenido renderizado en un canvas es opaco para el navegador: el texto no es seleccionable, las herramientas de accesibilidad no lo detectan, los buscadores no lo indexan y las herramientas de desarrollo no permiten inspeccionar elementos individuales. Compose HTML, por el contrario, genera elementos DOM reales (`<div>`, `<span>`, `<img>`, `<input>`), lo que preserva la accesibilidad, la inspección en DevTools, la selección de texto y el comportamiento natural del navegador (scroll, eventos de foco, formularios), a cambio de requerir estilos CSS explícitos en lugar de los modificadores de Compose Material.

Para un proyecto académico que necesita demostrar la interfaz ante un tribunal y cuya interfaz se compone de texto, imágenes y controles estándar, Compose HTML es la opción más defendible: el bundle producido es significativamente más pequeño (1,19 MiB frente a los cinco o seis del modo Canvas por la inclusión de Skiko), la experiencia del usuario es indistinguible de una aplicación web convencional, y las herramientas de depuración del navegador son plenamente operativas.

Se descartó también la opción de utilizar React, Vue o Angular para el frontend. Estas tecnologías habrían introducido un lenguaje adicional (TypeScript o JavaScript), habrían obligado a duplicar o generar los DTOs del backend, y habrían impedido demostrar una de las propiedades diferenciales de Kotlin Multiplatform: la capacidad de compartir modelos de datos entre servidor y cliente compilando el mismo código Kotlin a JVM y a JavaScript.

**Navegación sin biblioteca externa.** En aplicaciones web monopágina (SPA) es habitual recurrir a bibliotecas de enrutamiento que gestionan el mapeo entre la URL del navegador y la pantalla visible. En el ecosistema de Compose Multiplatform la biblioteca más referenciada es Voyager; sin embargo, al momento del desarrollo Voyager no soportaba el target Compose HTML (su compatibilidad se limita a Compose for Canvas, Android, iOS y escritorio). Ante esta incompatibilidad se optó por implementar un enrutamiento manual basado en **fragmentos de URL** (hash routing), donde la parte de la URL posterior al carácter `#` identifica la pantalla activa (`/#/movies`, `/#/movie/550`, `/#/search/matrix`). El hash nunca se envía al servidor, lo que elimina la necesidad de configurar un *catch-all* o *fallback* en Ktor para devolver `index.html` ante rutas desconocidas. El estado de la ruta se gestiona en un objeto singleton `AppState` con una propiedad observable `currentRoute` que Compose observa automáticamente para decidir qué pantalla renderizar; la integración con la API `History` del navegador (`pushState`, evento `popstate`) permite que los botones de avance y retroceso funcionen de forma nativa. El proyecto referente de la consultoría externa empleaba el mismo patrón con quince o más pantallas, lo que validó su escalabilidad para proyectos de mayor envergadura.

**CSS *custom properties* frente a constantes Kotlin.** El sistema de estilado se diseñó en torno a variables CSS declaradas en la pseudoclase `:root` y consumidas en todo el árbol de estilos mediante `var(--nombre)`. Este enfoque se eligió sobre la alternativa de definir los colores como constantes Kotlin y pasarlos como parámetros a los Composables por tres motivos: primero, la alternancia entre modo oscuro y modo claro se implementa como la adición o eliminación de una clase CSS en el elemento `<body>`, sin recomposición del árbol de Compose; segundo, las variables CSS son visibles y editables en tiempo real en las herramientas de desarrollo del navegador, lo que acelera enormemente la iteración visual durante el desarrollo; y tercero, los componentes individuales referencian tokens semánticos (`--text-primary`, `--bg-secondary`, `--accent`) en lugar de colores literales, estableciendo un contrato de diseño que facilita cambios globales de paleta sin modificar ningún componente.

### 7.3.5.3 Ejecución

La ejecución del bloque B7 siguió estrictamente el orden de sub-pasos planificado. Se documentan a continuación los aspectos más relevantes de cada sub-paso, con énfasis en las decisiones tomadas durante la implementación y en los resultados obtenidos.

**B7.1 — Centralización de versiones y módulo compartido multiplatforma.** Se creó el fichero `gradle/libs.versions.toml` centralizando las versiones de las trece dependencias del proyecto (Kotlin 2.1.20, Ktor 3.1.3, Compose 1.10.0, kotlinx-serialization 1.8.1, entre otras). Los tres ficheros `build.gradle.kts` (raíz, `:server`, `:composeApp`) pasaron a referenciar dependencias mediante *type-safe accessors* como `libs.ktor.server.netty` en lugar de cadenas literales con versión embebida. El módulo `:shared` se convirtió de `kotlin("jvm")` a `kotlin("multiplatform")` con targets `jvm()` y `js(IR) { browser() }`. Los ocho ficheros de DTOs se trasladaron de `src/main/kotlin/` a `src/commonMain/kotlin/` mediante `git mv`, preservando la similitud al cien por cien en el historial de Git. El fichero `PaginatedResponse.kt`, cuyo `package` declarado era `com.tfg.movies.shared.pagination` pero residía físicamente en el directorio `movies/`, se movió a su ubicación correcta dentro de `pagination/`; el cambio no requirió ninguna modificación en el código del servidor o el cliente porque todos los imports ya utilizaban el paquete declarado.

**B7.2 — Creación del módulo `:composeApp`.** Se registró un nuevo módulo en `settings.gradle.kts` y se configuró su `build.gradle.kts` con los plugins de Kotlin Multiplatform y Compose. La versión 1.10.0 de Compose Multiplatform deprecó los *accessors* `compose.runtime` y `compose.html.core`, lo que requirió apuntar las dependencias directamente a las coordenadas Maven de los artefactos a través del catálogo de versiones. Se añadió el repositorio de Google Maven (`google()`) en el `build.gradle.kts` raíz porque los artefactos de Compose se publican allí. Se configuró el fichero `src/jsMain/resources/index.html` con un elemento `<div id="root">` como punto de montaje del árbol de Compose y un script de carga del bundle `composeApp.js`. La compilación inicial del bundle de desarrollo se completó con éxito, produciendo un fichero de 12,4 MiB (no minificado, con *source maps* para depuración); el bundle de producción, generado por Webpack en modo de optimización, pesa 1,19 MiB.

**B7.3 — Cliente HTTP tipado.** Se implementaron cuatro ficheros en el paquete `api/`: `ApiException.kt` (treinta líneas, excepción con campos `status`, `errorCode` y `userMessage`), `ApiConfig.kt` (treinta y ocho líneas, resolución de URL base con autodetección de origen), `ApiClient.kt` (ciento diez líneas, configuración de `HttpClient(Js)` con `ContentNegotiation` y `Json`, método genérico `get<T>(path): Result<T>` con *reified type parameters* y construcción de *query strings* con filtrado de valores nulos) y `MovieApi.kt` (ciento veintisiete líneas, cuatro métodos de negocio más conversores de enums a parámetros de consulta). La verificación de la cadena completa se realizó con un botón temporal "Test API" en la interfaz que invocaba `MovieApi.getMovies(page = 1, pageSize = 5)` y mostraba el resultado en línea; la respuesta del backend llegó en veintisiete milisegundos con cinco películas correctamente deserializadas.

**B7.4 — Pantalla de listado.** El sub-paso más extenso del bloque se ejecutó en cinco fases internas. La primera fase definió el sistema de diseño en un objeto `AppStyle : StyleSheet()` que inyecta treinta y dos CSS *custom properties* en `:root`: tres capas de fondos oscuros (de `#0d1117` a `#21262d`), tres niveles de texto, un acento lavanda (`#a78bfa`), tokens de borde y sombra, una pila tipográfica de sistema, una escala de espaciados basada en múltiplos de cuatro píxeles, y dos radios de borde. Se incluyen resets base para `html`, `body`, encabezados, párrafos, enlaces y botones (con dos variantes: secundario transparente y primario con fondo de acento).

La segunda fase construyó el componente `MovieCard`, con estructura de póster (ratio 2:3 forzado por CSS `aspect-ratio`), título (una línea con truncamiento por `text-overflow: ellipsis`), fila de metadatos (año y valoración en color de acento) y hasta tres géneros renderizados como *pills*. Los pósteres se cargan directamente desde el CDN de TMDB en la variante `w300`; para películas cuyo `posterPath` es nulo o cuya imagen ya no existe en el CDN, un mecanismo de fallback basado en el atributo `onerror` del elemento `<img>` oculta la imagen rota y revela un hermano `<div>` con el emoji 🎬 y el título de la película. La detección de la expiración del póster fue una dificultad no anticipada que se documenta en el sub-apartado 7.3.5.4.

La tercera fase envolvió los componentes en un `MovieGrid` basado en CSS Grid con la regla `grid-template-columns: repeat(auto-fill, minmax(180px, 1fr))`, que produce automáticamente tantas columnas como quepan con un ancho mínimo de ciento ochenta píxeles, sin necesidad de *media queries*. Se validó el layout con diez películas mock en un fichero `MockData.kt`, incluyendo una entrada sin póster para ejercitar el fallback.

La cuarta fase reemplazó los datos mock por una llamada real a `MovieApi.getMovies()`. Se creó `MovieListScreen` con un modelo de estado basado en una `sealed class` con tres ramas (`Loading`, `Success`, `Error`), un `LaunchedEffect` para la carga inicial y un botón de reintento que incrementa un contador `reloadKey` observado por el efecto para relanzar la petición.

La quinta fase añadió la barra de filtros y el scroll infinito. Se implementó un nuevo endpoint `GET /genres` en el backend (función `findAllGenreNames()` en el repositorio, delegación en el servicio, ruta en `MovieRoutes`) que devuelve los veinte nombres de género de la base de datos ordenados alfabéticamente. En el frontend, el componente `MovieFilters` consume este endpoint al montarse y puebla un desplegable `<select>` con los géneros reales. Dos desplegables adicionales permiten elegir el campo de ordenación (popularidad, valoración, número de votos, fecha de estreno, título) y la dirección (ascendente, descendente). Los cambios de filtro incrementan un contador `filterKey` observado por un `LaunchedEffect` que vacía la lista de películas acumulada, resetea la página a uno y relanza la petición con los filtros actualizados.

El scroll infinito se implementó mediante un `DisposableEffect` que registra un *listener* en el evento `scroll` de `window`. Cuando el usuario se encuentra a menos de trescientos píxeles del final del viewport y no hay una carga en vuelo, se lanza una coroutine que solicita la página siguiente y concatena los resultados a la lista acumulada. Un flag `isLoadingMore` previene peticiones duplicadas por ráfagas de eventos de scroll. Un indicador "Cargando más películas…" aparece al final del grid mientras la petición está en vuelo. Un botón flotante circular con el color de acento, posicionado fijamente en la esquina inferior derecha y visible solo tras seiscientos píxeles de desplazamiento, permite retornar al inicio de la página con un clic.

**B7.5 — Pantalla de detalle y navegación.** Se introdujo `state/AppState.kt` como singleton observable con una propiedad `currentRoute` que determina qué pantalla renderiza `App()`. La navegación se realiza mediante `AppState.navigateTo(route)`, que actualiza la propiedad y llama a `window.history.pushState()` para modificar la barra de direcciones del navegador. Un *listener* del evento `popstate`, registrado al inicio de la aplicación, traduce las acciones de retroceso y avance del navegador en actualizaciones de la ruta, completando la integración bidireccional con la API `History`.

La pantalla de detalle (`MovieDetailScreen`) recibe el identificador de la película como parámetro y ejecuta dos peticiones al montar: `MovieApi.getMovie(id)` para la información completa y `MovieApi.getSimilarMovies(id, limit = 10, minVoteCount = 50)` para las recomendaciones semánticas. La disposición visual sigue un esquema de héroe con el póster a la izquierda (trescientos píxeles de ancho, ratio 2:3) y la información textual a la derecha (título, tagline en cursiva, metadatos, género). Debajo del héroe se muestran tres secciones: la sinopsis, un grid de los doce primeros miembros del reparto (nombre del actor y personaje interpretado), y una fila horizontal scrollable de tarjetas de películas similares. Cada tarjeta de la fila de similares es un `MovieCard` reutilizado, con un *handler* de clic que navega al detalle de esa película, permitiendo al usuario navegar entre películas relacionadas indefinidamente.

Los componentes `MovieCard` y `MovieGrid` se modificaron para aceptar un callback `onClick` opcional, de modo que la tarjeta puede ser presentacional (sin acción, como en la fila de similares del detalle cuando aún no se han implementado las interacciones) o interactiva (con navegación, como en el listado y los resultados de búsqueda).

**B7.6 — Búsqueda con autocompletado y barra de navegación.** El componente `NavBar` se renderiza en `App()` antes del despacho de pantallas, lo que lo hace visible en todas las rutas. Contiene el título "TFG Movies" como enlace al listado (ruta `movies`), un botón de alternancia de tema (sub-paso B7.7, documentado a continuación) y un campo de búsqueda con un botón "Buscar". El campo implementa autocompletado en tiempo real: un `LaunchedEffect` observa el texto introducido, espera trescientos milisegundos de inactividad (*debounce*), y si el texto tiene al menos dos caracteres, dispara una petición `MovieApi.searchMovies(query, pageSize = 8)`. Los resultados se muestran en un panel desplegable posicionado absolutamente debajo del campo, con cada sugerencia renderizada como una fila con miniatura del póster (cuarenta por sesenta píxeles), título, año, valoración y géneros. Un clic en una sugerencia navega al detalle y limpia el campo; la tecla Escape cierra el panel; la tecla Enter o el botón "Buscar" navegan a una pantalla de resultados completa (`SearchScreen`) con hasta cincuenta películas en formato grid.

El componente `SearchScreen` reutiliza `MovieGrid` y los mismos estados de carga, error y vacío definidos en `MovieListScreen`, con la diferencia de que no implementa scroll infinito: la búsqueda es un acto dirigido donde el usuario prefiere refinar su consulta antes que paginar indefinidamente resultados parcialmente relevantes.

**B7.7 — Modo claro y modo oscuro.** Se añadió al `StyleSheet` principal un bloque `.light-theme` que sobreescribe las variables de color con valores apropiados para fondos claros. El color de acento en modo claro se intensificó de `#a78bfa` (lavanda clara) a `#7c3aed` (púrpura profundo) para mantener un contraste suficiente sobre fondos blancos. El estado del tema se gestiona en `AppState` mediante una propiedad observable `isDarkTheme` y un método `toggleTheme()` que añade o elimina la clase `light-theme` del elemento `<body>` del DOM, persiste la preferencia en `localStorage` y actualiza la propiedad observable para que el botón de la barra de navegación refleje el icono correcto (sol en modo oscuro, luna en modo claro). Al iniciar la aplicación, `initTheme()` lee la preferencia almacenada; si no existe, consulta la *media query* `prefers-color-scheme: dark` del sistema operativo para respetar la configuración del usuario.

**B7.8 — Servir el frontend desde Ktor y configuraciones de ejecución.** Se registró en `server/build.gradle.kts` una tarea Gradle `copyFrontend` de tipo `Copy` que depende de `:composeApp:jsBrowserProductionWebpack` y copia los ficheros del bundle de producción (ficheros JavaScript desde `composeApp/build/kotlin-webpack/js/productionExecutable/` y el fichero `index.html` desde `composeApp/build/processedResources/js/main/`) al directorio `server/build/resources/main/static/`, donde quedan incluidos en el classpath del servidor. La tarea se conecta como dependencia de `classes`, de modo que cualquier build del servidor compila automáticamente el frontend si éste ha cambiado (Gradle detecta si los ficheros fuente están *up-to-date* y omite la recompilación en caso afirmativo). En `Application.kt` se añadió una única línea de configuración: `staticResources("/", "static")`, colocada después de las rutas de la API para que éstas tengan prioridad en la resolución.

Se modificó `ApiConfig.kt` para autodetectar si el frontend se ejecuta desde el mismo origen que la API: si el puerto de la página actual es 8081 (servidor de desarrollo Webpack), la URL base apunta a `http://localhost:8080`; en cualquier otro caso, se utiliza una cadena vacía, lo que produce peticiones con rutas relativas (`/movies` en lugar de `http://localhost:8080/movies`). Este mecanismo permite que el mismo bundle funcione tanto en modo de desarrollo (dos procesos) como en modo de producción (un solo proceso) sin recompilación.

Se añadió además una tarea `runDev` de tipo `JavaExec` que lee las variables de entorno del fichero `.env` del proyecto y las pasa al proceso Java, resolviendo una limitación del plugin Ktor que impide la correcta propagación de variables de entorno desde las configuraciones de IntelliJ al proceso *forked* por `./gradlew :server:run`. La tarea incluye la activación del modo de desarrollo de Ktor (`-Dio.ktor.development=true`) y depende tanto de `classes` como de `copyFrontend`, de modo que un solo clic en IntelliJ compila el frontend, lo copia y arranca el servidor con todas las variables configuradas.

Finalmente, se crearon cuatro configuraciones de ejecución de IntelliJ almacenadas en el directorio `.run/` del proyecto (y, por tanto, versionadas en Git): `Database (up)` y `Database (down)` como Shell Scripts que ejecutan `podman-compose up -d` y `podman-compose down` respectivamente; `Backend (Ktor)` como tarea Gradle `:server:runDev`; y `Frontend (dev)` como tarea Gradle `:composeApp:jsBrowserDevelopmentRun`.

### 7.3.5.4 Dificultades encontradas

El bloque B7 presentó cuatro dificultades técnicas significativas, tres de ellas no anticipadas en la planificación.

**Agotamiento de memoria del compilador Kotlin/JS.** Al incorporar las dependencias de Ktor Client y kotlinx-coroutines al módulo `:composeApp` en el sub-paso B7.3, el compilador Kotlin/JS comenzó a fallar con `java.lang.OutOfMemoryError: Java heap space` durante la fase de deserialización de las bibliotecas precompiladas (ficheros `.klib`). El daemon de Kotlin arranca por defecto con 512 MiB de heap, insuficientes para procesar el grafo de dependencias ampliado. La solución consistió en elevar el heap del daemon a 4 GiB (`kotlin.daemon.jvmargs=-Xmx4g` en `gradle.properties`) y el del daemon de Gradle a 2 GiB (`org.gradle.jvmargs=-Xmx2g`). Estas cifras son cómodas en el equipo de desarrollo, que dispone de 30 GiB de RAM. La dificultad se documenta como un recordatorio de que los proyectos Kotlin Multiplatform con Compose y Ktor requieren más memoria que los proyectos Kotlin/JVM puros, un detalle que las guías de inicio no siempre mencionan.

**Expiración de pósteres en el CDN de TMDB.** Durante la conexión del grid al backend real en B7.4, se observó que un número significativo de películas no mostraba su póster. La investigación inicial apuntó al mecanismo de seguridad Opaque Response Blocking (ORB) de Chrome, pero una verificación directa en el navegador confirmó que las imágenes afectadas devolvían un 404 ("File Not Found") incluso al acceder a la URL completa en una pestaña aislada. La causa resultó ser que las rutas de póster almacenadas en la base de datos proceden de un dataset estático de TMDB, y TMDB actualiza y elimina imágenes periódicamente. Las rutas eran válidas cuando se creó el dataset pero han caducado desde entonces. La solución en el frontend fue un mecanismo de degradación elegante: cada elemento `<img>` incluye un atributo `onerror` con un *handler* JavaScript inline que, al detectar un fallo de carga, oculta la imagen rota (`this.style.display='none'`) y revela un elemento hermano previamente oculto que muestra el emoji 🎬 y el título de la película. Este fallback es visualmente coherente con el grid y no rompe la disposición de las tarjetas. La dificultad se documenta honestamente como una **limitación inherente al uso de datasets estáticos** para datos cuyas fuentes externas cambian con el tiempo; una ampliación natural del proyecto consistiría en actualizar periódicamente los datos desde la API de TMDB con clave de acceso propia.

**Elección del color de acento.** La paleta inicial del tema oscuro utilizaba ámbar (`#f5a524`) como color de acento, elegido por su alto contraste sobre fondos carbón y su asociación visual con el cine (proyectores, luces cálidas). Sin embargo, durante la revisión visual se detectó que la combinación negro carbón con ámbar dorado es la paleta más reconocible de un conocido sitio para adultos, una asociación indeseable para un proyecto académico. Se iteró hacia un acento lavanda (`#a78bfa`), que mantiene el contraste necesario, no remite a ninguna marca o categoría problemática, no compite visualmente con los colores variados de los pósteres de cine, y sigue la tendencia estética de aplicaciones tecnológicas contemporáneas. Para el modo claro, se intensificó a un púrpura más profundo (`#7c3aed`) para compensar la pérdida de contraste sobre fondos blancos. La dificultad es menor en términos técnicos (un cambio de dos líneas) pero ilustra que las decisiones de diseño visual requieren validación contextual además de validación funcional.

**Propagación de variables de entorno en Gradle.** El sub-paso B7.8 reveló una limitación en la interacción entre el plugin Ktor de Gradle, el daemon de Gradle y las configuraciones de ejecución de IntelliJ. Al configurar la tarea `:server:run` (gestionada internamente por el plugin Ktor) como una configuración de tipo Gradle en IntelliJ, las variables de entorno definidas en el campo "Environment variables" de la configuración no llegaban al proceso Java *forked* por Gradle. Se probaron varias estrategias: ejecución sin daemon (`--no-daemon`), carga y exportación del fichero `.env` en el shell (`set -a && source .env && set +a`), y configuración programática de la tarea `run` mediante `tasks.named<JavaExec>("run") { environment(...) }`. Ninguna resolvió el problema de forma fiable. La solución definitiva fue crear una tarea Gradle propia (`runDev`) de tipo `JavaExec` con control explícito del classpath, la clase principal y las variables de entorno, estas últimas leídas directamente del fichero `.env` del proyecto durante la fase de configuración de Gradle. Esta tarea evita por completo la cadena de delegación del plugin Ktor y funciona de forma consistente tanto desde IntelliJ como desde terminal.

---

Con estas dificultades resueltas, el bloque B7 se cierra con una aplicación cliente completa que consume los cinco endpoints del backend (listado paginado, detalle, búsqueda, recomendaciones similares y listado de géneros). La arquitectura del frontend comprende diecisiete ficheros Kotlin organizados en cuatro paquetes (`api/`, `components/`, `screens/`, `state/`), más un fichero de constantes, un sistema de diseño y un punto de entrada. El bundle de producción pesa 1,19 MiB minificado y se sirve desde el propio servidor Ktor, produciendo un despliegue de proceso único que facilita la demostración ante el tribunal. El proyecto queda en condiciones de redactar la memoria final y, si el calendario lo permite, incorporar las ampliaciones opcionales documentadas en la sección 8.
